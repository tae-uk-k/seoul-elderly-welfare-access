# 서울시 복지 데드존 분석 — 데이터 전처리 과정 정리

> 이 문서는 분석에 쓴 데이터들을 어떻게 다듬었는지,  
> **어떤 문제가 있어서, 왜 못 쓰고, 어떻게 바꿨는지**를 기록한 문서입니다.  
> 개발을 몰라도 읽을 수 있도록 최대한 쉽게 썼습니다.

---

## 이 분석이 하려는 것

서울에서 **노인 복지시설이 있어도 실제로 못 가는 동네(복지 데드존)**를 찾는 것이 목표입니다.

가설은 이겁니다:
> **복지시설이 멀고, 버스·지하철도 안 다니는 동네일수록 → 노인들이 실제로 덜 돌아다닌다.**

이걸 데이터로 증명하려면 여러 종류의 데이터를 하나로 합쳐야 하는데, 처음 받은 데이터들은 제각각 문제가 있어서 그냥은 쓸 수 없었습니다.

## 문제 1. 복지시설에 지도 좌표가 없다

### 문제

복지시설 목록 파일에는 `서울특별시 강남구 역삼1동 823-1` 같은 주소는 있었지만, 지도에 올리거나 거리를 계산하려면 **위도·경도 좌표**가 필요합니다.

주소만으로는 "이 시설이 저 동네에서 몇 km 떨어져 있나?"를 계산할 수 없습니다.

### 어떻게 바꿨나

**카카오 지도 API**를 이용해 주소 하나하나를 좌표로 변환했습니다.

> 쉽게 비유하면, 사람이 카카오 지도 검색창에 주소를 직접 치는 것을 프로그램이 3,126번 자동으로 반복한 것입니다.

---

#### 1차 시도 — 주소 정제 후 변환 (3,126건 전체)

API에 넣기 전에 주소를 먼저 다듬었습니다. 원본 주소에 불필요한 내용이 섞여 있으면 검색이 실패하기 때문입니다.

| 문제 있는 주소 | 정제 후 |
|--------------|--------|
| `청담동(삼성동) 15-3` | 괄호 안 제거 → `청담동 15-3` |
| `강남구  대치동` (공백 두 칸) | 공백 정리 → `강남구 대치동` |
| `강서구 화곡1동` ('서울특별시' 없음) | 앞에 자동 추가 → `서울특별시 강서구 화곡1동` |

그래도 검색이 안 되는 경우에는 주소 대신 **"서울특별시 강남구 OO노인복지관"** 처럼 시설명으로 재검색했습니다.

혹시 API가 엉뚱한 좌표(예: 서울이 아닌 지방 위치)를 주는 경우를 막기 위해, 서울 경계를 벗어나는 좌표는 자동으로 걸러냈습니다.

1차 시도 결과: 전체 3,126건 중 약 204건 실패 → 실패 목록을 따로 저장

---

#### 2차 시도 — 실패한 204건 강화 정제 후 재시도

실패한 주소들을 열어보니 공통적인 패턴이 있었습니다. 주소에 층수, 호수, 빌딩 이름 등이 붙어 있어서 API가 제대로 인식하지 못한 것입니다.

| 문제 패턴 | 예시 | 처리 방법 |
|----------|------|----------|
| 층수 표기 | `역삼로 23 3,4층` | 층수 부분 제거 → `역삼로 23` |
| 호실 번호 | `도림로 7동104호` | 호실 제거 → `도림로 7동` |
| 지하층 복합 표기 | `세종대로 B5층` | 제거 |
| 대괄호 안 설명 | `홍은제1동[홍은동] 12` | 대괄호 제거 → `홍은제1동 12` |
| 숫자만 있는 오류 데이터 | `12345` | 애초에 주소가 아님 → 스킵 |

2차까지 시도한 후에도 못 찾은 건은 `지오코딩_최종실패.xlsx`에 별도 보관했습니다.

**결과**: 대부분의 시설 좌표 확보 완료 → `서울_노인복지시설_좌표.xlsx` 저장

## 문제 2. 독거노인 데이터에 동네 코드가 없다

### 문제

독거노인 현황 파일에는 동네 이름이 **텍스트**로만 적혀 있었습니다.

```
개포1동  120명
신사동   95명
역삼1동  200명
```

문제는 데이터끼리 합칠 때 이름 텍스트로 연결하면 오류가 납니다.  
예를 들어 서울에도 "신사동"이 있고 경기도에도 비슷한 이름이 있을 수 있습니다. 또 파일마다 "역삼1동" vs "역삼동" 처럼 표기가 달라도 서로 다른 동으로 인식합니다.

공식 행정동 코드(예: `11680510`)가 있어야 혼동 없이 다른 데이터와 정확히 연결됩니다.

### 어떻게 바꿨나

공식 행정동 지도 파일(`BND_ADM_DONG_PG.shp`)에는 동 이름과 행정동 코드가 함께 들어 있습니다. 이 파일의 동 이름과 독거노인 파일의 동 이름을 매칭해서 코드를 가져왔습니다.

그런데 바로 이름으로 비교하면 또 문제가 생깁니다. 파일마다 표기가 미묘하게 다르기 때문입니다.

```
지도 파일: "역삼1동"
독거노인 파일: "역삼1동 (구역삼동)"
→ 그냥 비교하면 다른 것으로 인식
```

그래서 두 파일 모두 **한글, 숫자, 영문만 남기고 나머지를 전부 지운** 뒤 비교했습니다.

```
"역삼1동 (구역삼동)" → 괄호·공백·특수문자 제거 → "역삼1동"
"역삼1동"           → 그대로                   → "역삼1동"
→ 이제 같은 것으로 인식 ✅
```

**결과**: 독거노인_행정동코드포함.csv — 행정동 코드가 추가된 파일 생성

## 문제 3. 65세 이상 인구가 구(자치구) 단위밖에 없다

### 문제

처음에 확보한 인구 파일(`연령별_인구현황_자치구_2026년3월.xlsx`)은 **강남구 전체** 인구만 알려줍니다. 강남구 안에서도 어느 동네에 노인이 많은지는 알 수 없습니다.

분석은 **동네 단위**로 해야 의미 있는 데드존을 찾을 수 있는데, 구 단위로는 너무 뭉뚱그려져 있습니다.

### 어떻게 바꿨나

**동네(행정동) 단위로 집계된 별도 파일**(`연령별_인구현황_서울_2026년4월.csv`)을 추가로 확보했습니다. 이 파일에는 행정동별로 나이대가 하나하나 컬럼으로 나뉘어 있었습니다.

이 컬럼들 중 65세부터 100세 이상까지 해당하는 컬럼들을 모두 더해서 행정동별 65세 이상 합계를 만들었습니다.

```
[개포1동] 65세 120명 + 66세 108명 + ... + 100세이상 3명 = 65세이상 합계
[역삼1동] 65세 200명 + 66세 185명 + ... + 100세이상 5명 = 65세이상 합계
(모든 행정동에 대해 반복)
```

주의한 점: 이 파일에는 나이 컬럼이 0세부터 100세 이상까지 빼곡히 들어 있고, 100세 이상 다음 컬럼부터는 전혀 다른 데이터(합계 등)가 시작됩니다. 범위를 잘못 잡으면 엉뚱한 숫자가 섞이기 때문에, 65세 컬럼의 위치를 먼저 찾고 거기서 100세 이상까지만 더했습니다.

**결과**: 행정동별_65세이상인구.csv 생성

## 문제 4. 생활인구 데이터에 새벽·심야가 섞여 있다

### 문제

생활인구 데이터(`생활인구_행정동별_시간대별_2026년4월.csv`)는 424개 행정동 × 24시간을 모두 기록합니다. 그래서 행이 엄청나게 많습니다(305,280행).

그냥 모든 시간대를 평균 내면 이런 일이 생깁니다:

- 새벽 3시: 노인 생활인구 거의 0명 (다들 집에서 자고 있음)
- 낮 2시: 노인 생활인구 많음

이 둘을 그냥 평균 내면 "낮에 노인이 많다"는 신호가 약해집니다. 복지시설 접근성이 좋아도 새벽엔 아무도 안 다니니, 접근성 효과가 데이터에서 사라지는 것입니다.

### 어떻게 바꿨나

**오전 9시부터 오후 5시 사이 데이터만** 골라서 사용했습니다. 노인이 실제로 외출할 수 있는 시간대입니다.

```
새벽~아침(0~8시) 데이터 제거
저녁~심야(18~23시) 데이터 제거
주간(9~17시) 데이터만 남겨서 동네별 평균 계산
```

이렇게 하니 생활인구와 접근성의 관계가 훨씬 명확하게 드러났습니다.

## 문제 6. 지도 파일과 다른 데이터의 좌표 단위가 다르다

### 문제

거리를 계산하려면 모든 데이터의 위치 표현 방식이 같아야 합니다.  
그런데 데이터마다 좌표 체계가 달랐습니다.

| 데이터 | 좌표 체계 | 설명 |
|--------|----------|------|
| 버스정류소, 지하철역, 복지시설 | WGS84 | 위도 37.xx, 경도 126.xx 형태 |
| 서울 250m 격자 지도 파일 | EPSG:5179 (TM 좌표계) | 미터 단위 숫자 |
| SGIS 100m 격자 지도 파일 | EPSG:5179 (TM 좌표계) | 마찬가지로 미터 단위 숫자 |

**WGS84 좌표로 거리를 계산하면 안 됩니다.** 위도·경도는 각도 단위이기 때문에 그 차이가 몇 미터인지 바로 알 수 없습니다. "위도 0.001도 차이"가 서울에서 몇 미터인지 별도 계산이 필요합니다.

EPSG:5179는 미터 단위이므로 좌표 차이가 곧 미터 거리가 됩니다.

### 어떻게 바꿨나

거리 계산에 사용하는 모든 데이터를 **EPSG:5179(미터 단위)**로 통일 변환했습니다.

```
버스정류소 WGS84 → EPSG:5179 변환
지하철역   WGS84 → EPSG:5179 변환
복지시설   WGS84 → EPSG:5179 변환
(격자 지도는 원래 EPSG:5179)
```

이 변환 후에야 "복지관까지 거리 몇 미터", "반경 500m 안 버스정류소 몇 개"를 정확히 계산할 수 있었습니다.

## 문제 7. 독거노인 엑셀이 사람이 읽기 편한 형태로 저장되어 있다

### 문제

원본 독거노인 파일은 사람이 눈으로 보기 좋게 만들어진 엑셀입니다.

```
[엑셀 구조 예시]
1행: (큰 제목)
2~4행: (소제목 / 설명)
5행: 서울특별시 | 강남구 | 소계 | 1,234명 | ...
6행:             |        | 개포1동 | 120명 |
7행:             |        | 개포2동 |  95명 |
8행: 서울특별시 | 강동구 | 소계 | ...
```

이 구조의 문제:
- 구 이름(강남구)이 소계 행에만 쓰여 있고, 아래 동 행에는 비어 있음
- 처음 4행은 타이틀·헤더라 데이터가 아님
- 숫자에 쉼표가 들어가 있음(`1,234` → 컴퓨터는 문자열로 읽음)

### 어떻게 바꿨나

1. 처음 4행을 건너뛰고 5행부터 데이터로 읽음
2. 구 이름이 빈 행에는 바로 위 행의 구 이름을 내려받아 채움  
   → `강남구 | 개포1동`, `강남구 | 개포2동` 형태로 정리
3. 소계 행은 분석에서 제외 (동별 데이터만 사용)
4. `1,234` → 쉼표 제거 → `1234`로 변환
5. 빈칸(`-`)은 0으로 처리

## 문제 8. 구 단위로는 분석이 너무 뭉뚱그려진다

### 문제

처음에는 자치구(25개 구) 단위로 분석을 시작했습니다.  
그런데 "복지 접근성이 낮은 곳에서 노인이 덜 돌아다닌다"는 가설을 검증하니, 통계적으로 아무 관계가 없다는 결과가 나왔습니다 (상관계수 -0.003).

이유가 있었습니다. **구는 너무 크기 때문**입니다. 강남구 안에도 복지시설이 많은 동네와 적은 동네가 섞여 있는데, 구 전체를 평균 내면 그 차이가 사라집니다.

### 어떻게 바꿨나

분석 단위를 **자치구 → 행정동(약 424개 동) → 100m 격자(약 60,528개 격자)**로 점점 세밀하게 쪼갰습니다.

격자는 서울 전체를 100m × 100m 격자로 나눈 것입니다. 동네(행정동)보다 훨씬 촘촘하고, 같은 동네 안에서도 어느 블록이 복지 사각지대인지 볼 수 있습니다.

---

#### 핵심: 100m 격자 인구 데이터를 별도로 다운받았다

행정동 단위 생활인구를 격자에 배분하려면, 격자마다 실제로 사람이 얼마나 사는지를 알아야 합니다. 이를 위해 **SGIS(통계지리정보서비스)에서 100m 격자별 인구 데이터**를 별도로 다운받았습니다.

| 파일 | 내용 |
|------|------|
| `2024년_인구_다사_100M.csv` | 전국 100m 격자별 총인구 (서울: 60,528개 격자) |
| `grid_다사_100M.shp` | 100m 격자 경계 지도 파일 |

이 데이터 덕분에 "역삼1동 안에서도 격자 A는 총인구 500명, 격자 B는 총인구 50명"이라는 사실을 알 수 있었습니다.

---

#### 인구 가중치 비율로 배분

이 격자별 총인구를 **가중치(비율)**로 사용해서, 행정동 생활인구를 격자별로 나눴습니다.

```
역삼1동 전체 65세이상 생활인구: 300명
  ├─ 격자 A (총인구 500명 / 동 전체 1,000명 = 50%)  → 300 × 50% = 150명 배분
  ├─ 격자 B (총인구 300명 / 동 전체 1,000명 = 30%)  → 300 × 30% =  90명 배분
  └─ 격자 C (총인구 200명 / 동 전체 1,000명 = 20%)  → 300 × 20% =  60명 배분
```

이전 250m 격자 분석에서는 사람이 실제로 얼마나 사는지 정보가 없어서 격자 수로 균등 배분했습니다. 100m 격자에서는 실제 인구 비율을 반영해 더 현실적으로 배분할 수 있었습니다.

예외 처리: 행정동 내 총인구가 0으로 기록된 격자가 있으면 균등 배분으로 대체했습니다.

## 문제 9. 복지시설 거리·버스·지하철을 따로 넣으면 통계가 망가진다

### 문제

처음 회귀분석에서 "복지관까지 거리", "버스정류소 수", "지하철역까지 거리" 세 변수를 따로 넣었습니다.  
그런데 세 변수 모두 통계적으로 무의미하다는 결과가 나왔고, VIF(다중공선성 지표)가 12.66으로 매우 높게 나왔습니다.

**다중공선성 문제**란 쉽게 말해 이렇습니다:

> 도심 지역은 복지관도 가깝고, 버스도 많고, 지하철도 가깝습니다.  
> 외곽 지역은 복지관도 멀고, 버스도 적고, 지하철도 멉니다.
>
> 즉, 세 변수가 사실상 같은 정보(도심이냐 외곽이냐)를 중복으로 담고 있는 것입니다.  
> 이 상태에서 회귀분석을 하면 "복지관이 영향을 주는 건지, 버스가 영향을 주는 건지" 구분이 안 됩니다.

### 어떻게 바꿨나

**PCA(주성분 분석)**라는 방법으로 세 변수를 **"종합 접근성 지수" 하나**로 합쳤습니다.

> 비유하자면: 키, 몸무게, BMI를 따로 넣는 대신 "체격"이라는 하나의 지표로 합치는 것입니다.

"복지관 거리 + 버스 적음 + 지하철 거리"를 합쳐서 **값이 클수록 접근성이 나쁜** 하나의 숫자로 만들었습니다. 이 상태로 분석하니 다중공선성이 해결됐습니다.
